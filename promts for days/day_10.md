# Единый ручной промпт: sliding window + sticky facts + branching

## Цель

Один сценарий ручного тестирования, который позволяет проверить:

- сохранение sticky facts;
- поведение sliding window;
- создание checkpoint и branching;
- изоляцию изменений между ветками.

## Настройки агента

- Режим контекста: Скользящее окно
- Окно памяти: 3
- Summary: любое значение, не должно влиять на тест

## Стартовый промпт

Отправь агенту это сообщение первым сообщением в новом чате:

```text
Запомни и дальше учитывай следующие постоянные факты о проекте.

Проект называется Nova Desk.
Нужно подготовить лендинг для запуска курса по Python для аналитиков.
Главная цель - собрать 300 заявок до 15 августа 2026.
Бюджет на рекламу - не больше 120 000 рублей.
Не используем агрессивные продажи и не обещаем трудоустройство.
Отвечай кратко, списками, без длинных вступлений.
Сначала делаем MVP на Tilda.
Отдельный сайт делаем только если конверсия будет выше 7%.
Контактное лицо - Марина Соколова.
Telegram контакта - @marina_course.
Приоритет релиза - 12 июля.
Целевая аудитория - B2B-команды продаж.

Ответь только: записал.
```

## Основной сценарий

1. Отправить стартовый промпт и получить ответ `записал.`
2. Отправить: `Факт A: кодовый цвет проекта - янтарный. Ответь только: принято A.`
3. Отправить: `Факт B: кодовый город проекта - Казань. Ответь только: принято B.`
4. Отправить: `Факт C: кодовый фрукт проекта - груша. Ответь только: принято C.`
5. Нажать `Сохранить ветку отсюда`.
6. Отправить: `Факт D: кодовый инструмент проекта - компас. Ответь только: принято D.`
7. Отправить: `Перечисли все кодовые факты, которые доступны тебе из текущего контекста. Отдельно кратко напомни постоянные факты проекта.`

## Ожидание после основного сценария

- В prompt последнего запуска по sliding window должны попасть только последние 3 сообщения: про факт C, факт D и текущий вопрос.
- Агент не должен уверенно перечислять факты A и B как доступные из текущего короткого окна.
- Агент должен помнить sticky facts из стартового промпта:
  - название `Nova Desk`;
  - цель `300 заявок до 15 августа 2026`;
  - бюджет `не больше 120 000 рублей`;
  - тон `без агрессивных продаж и без обещаний трудоустройства`;
  - формат ответа `кратко, списками, без длинных вступлений`;
  - `MVP на Tilda`;
  - отдельный сайт только при конверсии выше `7%`;
  - контакт `Марина Соколова`;
  - telegram `@marina_course`;
  - приоритет релиза `12 июля`;
  - аудитория `B2B-команды продаж`.

## Проверка branching

1. Открыть список checkpoint и убедиться, что создана одна контрольная точка.
2. Создать из неё ветку `Ветка A`.
3. Создать из неё ветку `Ветка B`.
4. Убедиться, что обе ветки видны в списке чатов.
5. Убедиться, что в обеих ветках одинаковая история до checkpoint.
6. Убедиться, что в новых ветках нет сообщений, отправленных после checkpoint.

## Проверка изоляции веток

### Ветка A

Отправить по очереди:

1. `В этой ветке обновление: кодовый фрукт проекта теперь манго. Ответь только: обновил фрукт.`
2. `В этой ветке контактное лицо теперь Олег Миронов, telegram: @oleg_ops. Ответь только: обновил контакт.`
3. `Назови кодовый фрукт, контактное лицо и telegram для этой ветки. Затем кратко напомни название проекта.`

Ожидание:

- Ветка A отвечает: фрукт `манго`.
- Ветка A отвечает: контакт `Олег Миронов`.
- Ветка A отвечает: telegram `@oleg_ops`.
- Ветка A сохраняет общее sticky knowledge, например название `Nova Desk`.

### Ветка B

Отправить по очереди:

1. `В этой ветке обновление: кодовый фрукт проекта теперь слива. Ответь только: обновил фрукт.`
2. `В этой ветке бюджет теперь не больше 150 000 рублей. Ответь только: обновил бюджет.`
3. `Назови кодовый фрукт и лимит бюджета для этой ветки. Затем кратко напомни контактное лицо и telegram.`

Ожидание:

- Ветка B отвечает: фрукт `слива`.
- Ветка B отвечает: бюджет `не больше 150 000 рублей`.
- Ветка B не должна подтягивать изменения из ветки A.
- Ветка B должна сохранить свои копии остальных sticky facts, если они не менялись в этой ветке.

## Финальная перекрёстная проверка

1. Повторно открыть `Ветка A` и спросить: `Напомни фрукт, бюджет и контакт этой ветки.`
2. Повторно открыть `Ветка B` и спросить: `Напомни фрукт, бюджет и контакт этой ветки.`

## Финальное ожидание

- Ветка A изолированно помнит свои изменения: `манго`, контакт `Олег Миронов`, telegram `@oleg_ops`.
- Ветка A не должна получить бюджет `150 000 рублей` из ветки B без отдельного обновления.
- Ветка B изолированно помнит свои изменения: `слива`, бюджет `150 000 рублей`.
- Ветка B не должна получить контакт `Олег Миронов` или фрукт `манго` из ветки A.
- Обе ветки должны сохранять общие sticky facts из исходной точки ветвления, если эти факты не были переопределены внутри конкретной ветки.

## Что смотреть в trace

- Какие сообщения реально попали в prompt каждого запуска.
- Сохраняются ли sticky facts отдельно от короткого окна сообщений.
- Одинаков ли базовый контекст у `Ветки A` и `Ветки B` в момент создания.
- Не перетекают ли обновлённые факты между ветками.
