# Ответы

# Для температуры 0
Исходя из описания, у вас есть три ключевых компонента: **клиент** (создает серт), **OIDC-провайдер** (выпускает токены, привязанные к серту) и **Resource Server** (валидирует токены и mTLS).

В первую очередь нужно тестировать **критический путь аутентификации и валидации**, так как ошибки здесь приведут к полному отказу в доступе или, что хуже, к дырам в безопасности.

Вот приоритетные тест-кейсы:

---

### Тест кейс 1: Успешная валидация `cnf` (Confirmation) в токене
**Зачем тестируем:**
Это основная фича RFC 8705. Если Resource Server не проверяет, что `X.509` сертификат, предъявленный при mTLS, совпадает с хешем в токене (`x5t#S256`), то любой, кто украдет токен, сможет им воспользоваться. Это самый важный тест безопасности.

**Что проверить:**
1. Клиент получает токен от OIDC (токен содержит `cnf` с `x5t#S256`).
2. Клиент делает запрос к Resource Server, используя **правильный** сертификат (тот же, что был привязан).
3. **Ожидаемый результат:** HTTP 200, доступ разрешен.

---

### Тест кейс 2: Отказ при несовпадении сертификата и токена (Replay Attack)
**Зачем тестируем:**
Проверяем, что система защищена от кражи токена. Злоумышленник может перехватить Bearer-токен, но у него нет оригинального клиентского сертификата.

**Что проверить:**
1. Клиент А получает токен с привязкой к своему серту.
2. Клиент Б (или злоумышленник) пытается использовать этот токен, но предъявляет **другой** (свой) сертификат при mTLS.
3. **Ожидаемый результат:** HTTP 401 Unauthorized. Resource Server должен вернуть ошибку, так как хеш сертификата не совпадает с `cnf` в токене.

---

### Тест кейс 3: Валидация цепочки сертификатов (CA) на Resource Server
**Зачем тестируем:**
Если Resource Server принимает любой самоподписанный сертификат, это дыра. Нужно убедиться, что сервер доверяет только сертификатам, выпущенным конкретным CA (Certificate Authority), который вы контролируете.

**Что проверить:**
1. Клиент использует сертификат, подписанный **неизвестным** (недоверенным) CA.
2. Клиент предъявляет этот сертификат при mTLS.
3. **Ожидаемый результат:** HTTP 401 или 403 (TLS handshake может упасть еще на уровне транспорта, или сервер вернет ошибку на уровне приложения). Доступ запрещен.

---

### Тест кейс 4: Истечение срока действия сертификата (Expired Certificate)
**Зачем тестируем:**
Проверяем корректную обработку жизненного цикла сертификата. Если сервер принимает просроченный серт, это нарушает политику безопасности.

**Что проверить:**
1. Клиент использует сертификат с истекшим сроком действия (или сдвинуть время на клиенте/сервере).
2. Клиент пытается получить токен или обратиться к Resource Server.
3. **Ожидаемый результат:** Ошибка на этапе TLS handshake (обычно) или HTTP 401. Система не должна работать с просроченными кредами.

---

### Тест кейс 5: Отсутствие сертификата при запросе (Plain HTTPS)
**Зачем тестируем:**
Проверяем, что Resource Server не работает в "fallback" режиме. Если эндпоинт требует mTLS, а клиент приходит без сертификата, сервер должен отказать, а не пустить по обычному HTTPS.

**Что проверить:**
1. Клиент делает запрос к Resource Server **без** клиентского сертификата (обычный HTTPS).
2. В запросе передается валидный Bearer-токен (с привязкой).
3. **Ожидаемый результат:** HTTP 401 или 403. Сервер должен требовать mTLS.

---

### Рекомендуемый порядок выполнения (MVP):
1. **Тест 1** (Успех) — чтобы убедиться, что базовая механика работает.
2. **Тест 2** (Несовпадение) — самый важный негативный тест безопасности.
3. **Тест 5** (Без серта) — проверка принудительного mTLS.
4. **Тест 3** (CA) — проверка доверия к корневому центру.
5. **Тест 4** (Срок) — проверка жизненного цикла.

# Для температуры 0.7
Исходя из описания, у вас есть три ключевых компонента: **клиент** (создает серт), **OIDC-провайдер** (выпускает токены, привязанные к серту), и **Resource Server** (проверяет привязку). Самая критичная уязвимость в mTLS-bound tokens — это возможность украсть токен и использовать его с другого устройства (без оригинального серта).

Поэтому тестировать в первую очередь нужно **механизм привязки (binding)** и **проверку этой привязки**.

Вот приоритетные тест-кейсы (от самых критичных к менее критичным):

### 1. Тест кейс: Replay-атака с другим сертификатом
**Зачем тестируем:** Это основная защита, которую дает RFC 8705. Если этот тест провален, вся система не имеет смысла.
*   **Действие:** Получить Access Token (с `cnf` claim) через OIDC с использованием сертификата А.
*   **Действие:** Попытаться использовать этот же токен для доступа к Resource Server, но установив mTLS-соединение с сертификатом Б (другим, валидным, но не тем, который был привязан).
*   **Ожидание:** Resource Server должен вернуть **401 Unauthorized** (или 403). Токен должен быть отвергнут, так как `X5t#S256` в токене не совпадает с хешем сертификата Б.

### 2. Тест кейс: Replay-атака без mTLS (обычный HTTPS)
**Зачем тестируем:** Проверяем, что Resource Server не принимает токены, если клиент не прошел mTLS-аутентификацию вообще.
*   **Действие:** Получить Access Token через OIDC (с mTLS).
*   **Действие:** Отправить запрос к Resource Server **без** клиентского сертификата (обычный HTTPS-запрос, только с заголовком `Authorization: Bearer <token>`).
*   **Ожидание:** Resource Server должен вернуть **401 Unauthorized**. Если он принимает токен без mTLS, то токен можно украсть и использовать где угодно.

### 3. Тест кейс: Проверка формата `cnf` (Confirmation Claim)
**Зачем тестируем:** OIDC-провайдер должен правильно формировать claim. Если он положит туда неправильный хеш или не положит его вовсе, Resource Server не сможет проверить привязку.
*   **Действие:** Получить Access Token.
*   **Действие:** Декодировать JWT (base64url) и проверить наличие и структуру поля `cnf`.
*   **Ожидание:** В payload токена должен быть объект `"cnf": { "x5t#S256": "<base64url-encoded SHA-256 thumbprint>" }`. Хеш должен соответствовать публичной части сертификата, который был использован при получении токена.

### 4. Тест кейс: Истечение срока жизни токена (Token Expiry)
**Зачем тестируем:** Даже с привязкой к серту, если токен живет вечно, это проблема. Но в контексте mTLS это особенно важно, так как сертификат клиента может быть отозван, а токен останется.
*   **Действие:** Получить токен. Дождаться его истечения (или выставить короткий `exp` в настройках OIDC).
*   **Действие:** Попытаться использовать просроченный токен с правильным сертификатом.
*   **Ожидание:** Resource Server должен вернуть **401 Unauthorized** с указанием на истекший токен (обычно `invalid_token`).

### 5. Тест кейс: Использование токена с другим Resource Server (если их несколько)
**Зачем тестируем:** Проверка, что токен не является "золотым ключом" ко всем API.
*   **Действие:** Получить токен для Resource Server A.
*   **Действие:** Попытаться использовать этот токен (с тем же сертификатом) для доступа к Resource Server B (который не указан в `aud` токена).
*   **Ожидание:** Resource Server B должен вернуть **401 Unauthorized** (или 403), так как `aud` (audience) не совпадает.

### Итоговая рекомендация
Начните с **Тест кейса 1** и **Тест кейса 2**. Если эти два теста проходят, система выполняет свою основную функцию безопасности. Если нет — остальные тесты не имеют смысла, так как базовая защита сломана.

# Для темпаратуры 1.2
Исходя из описания функционала (mTLS Access Token Bound по RFC 8705), приоритет тестирования должен быть направлен на **безопасность привязки токена к сертификату** и **корректность валидации на Resource Server**. Это критические точки, где ошибка приведет к компрометации всей схемы.

Вот приоритетные тест-кейсы (от наиболее критичного к менее):

### Тест кейс 1: Валидация `cnf` (Confirmation) в токене
**Зачем тестируем:** Это основа RFC 8705. Если Resource Server не проверяет, что `x5t#S256` (SHA-256 thumbprint сертификата) в JWT совпадает с сертификатом, предъявленным при запросе, то любой, кто украдет токен, сможет его использовать с любым своим сертификатом. Это главная уязвимость.

### Тест кейс 2: Отказ при несовпадении thumbprint
**Зачем тестируем:** Проверяем негативный сценарий. Клиент предъявляет валидный токен, но использует **другой** валидный сертификат (например, от другого пользователя). Resource Server должен вернуть `401 Unauthorized`. Если этого не происходит — система небезопасна.

### Тест кейс 3: Отказ при отсутствии сертификата клиента (mTLS)
**Зачем тестируем:** Проверяем, что Resource Server требует mTLS. Если клиент присылает валидный токен, но соединение установлено без клиентского сертификата (обычный TLS), сервер должен отклонить запрос. Иначе токен можно использовать без mTLS, что нарушает всю модель безопасности.

### Тест кейс 4: Генерация и ротация сертификата клиента
**Зачем тестируем:** Проверяем жизненный цикл. Если клиент генерирует новый сертификат (например, при переустановке приложения), старый токен (с привязкой к старому сертификату) должен стать невалидным. Это проверяет, что `cnf` действительно привязан к конкретному экземпляру сертификата, а не к пользователю.

### Тест кейс 5: OIDC flow с mTLS (Token Endpoint)
**Зачем тестируем:** Проверяем, что Authorization Server (AS) при выдаче токена корректно извлекает thumbprint сертификата из mTLS-соединения и вшивает его в `cnf` JWT. Если AS использует сертификат из другого источника (например, из заголовка, а не из TLS), это может быть подделано.

### Тест кейс 6: Использование просроченного или отозванного сертификата
**Зачем тестируем:** Проверяем валидацию цепочки сертификатов. Если клиент использует просроченный сертификат, но токен еще жив, Resource Server должен отклонить запрос на уровне TLS (до проверки токена). Это проверяет корректность настройки TLS-терминации.

### Тест кейс 7: Повторное использование токена с другого IP/устройства (с тем же сертификатом)
**Зачем тестируем:** Проверяем, что привязка идет именно к сертификату, а не к сессии. Если злоумышленник украл и токен, и сертификат (например, с устройства), он сможет использовать их с другого IP. Это не баг, а особенность модели, но тест нужен для документирования поведения и оценки рисков.

### Рекомендуемый порядок выполнения:
1. **Тест 2** (несовпадение thumbprint) — самый быстрый и критичный.
2. **Тест 3** (отсутствие mTLS) — проверка базового требования.
3. **Тест 1** (валидация `cnf`) — проверка логики парсинга JWT.
4. **Тест 5** (OIDC flow) — проверка генерации токена.
5. **Тест 4** (ротация) — проверка жизненного цикла.
6. **Тест 6** (просрочка) — проверка TLS.
7. **Тест 7** (повторное использование) — документирование.

## Для температуры 2
Для задачи тестирования реализации mTLS Access Token Bound (RFC 8705) в первую очередь необходимо сфокусироваться на критических аспектах безопасности и корректности связывания токена с сертификатом. Вот приоритетные тест-кейсы:

## Тест кейс 1: Валидация привязки токена к сертификату (cnf/thumprint)
**Зачем тестируем:** Это основная защита от кражи токена. Если злоумышленник украдет access token, он не сможет его использовать без соответствующего сертификата. Проверяем, что сервер действительно проверяет соответствие `cnf` (confirmation) в JWT с отпечатком сертификата клиента.

## Тест кейс 2: Отказ при использовании чужого сертификата
**Зачем тестируем:** Проверяем, что система не пропускает запросы, где сертификат не соответствует токену. Это критично для безопасности - если этот кейс не работает, вся защита mTLS теряет смысл.

## Тест кейс 3: Проверка цепочки сертификатов (валидация CA)
**Зачем тестируем:** Убеждаемся, что сервер принимает только сертификаты от доверенного CA. Если пропустить самоподписанный сертификат или сертификат от неизвестного CA, злоумышленник сможет подделать привязку.

## Тест кейс 4: Обработка истечения срока сертификата
**Зачем тестируем:** Проверяем корректную обработку expired сертификатов. Сервер должен отклонять запросы с просроченными сертификатами, даже если токен еще действителен.

## Тест кейс 5: Валидация формата cnf в JWT
**Зачем тестируем:** Проверяем, что сервер корректно парсит и валидирует поле `cnf` (x5t#S256) в JWT. Некорректный формат или отсутствие этого поля должно приводить к ошибке.

## Тест кейс 6: Повторное использование токена с другим сертификатом
**Зачем тестируем:** Проверяем, что один и тот же токен нельзя использовать с разными сертификатами (даже от одного пользователя). Это предотвращает атаки переиспользования токена.

## Тест кейс 7: Обработка отсутствия сертификата в запросе
**Зачем тестируем:** Проверяем, что resource server корректно отклоняет запросы без mTLS сертификата, когда ожидается bound access token.

## Тест кейс 8: Валидация алгоритма хеширования в cnf
**Зачем тестируем:** RFC 8705 требует использования SHA-256 для отпечатка сертификата. Проверяем, что сервер не принимает другие алгоритмы (например, SHA-1) или некорректные значения.

## Тест кейс 9: OIDC flow с mTLS - получение bound token
**Зачем тестируем:** Проверяем полный цикл: клиент с сертификатом получает access token, bound к этому сертификату. Важно убедиться, что OIDC провайдер корректно формирует `cnf` в JWT.

## Тест кейс 10: Обработка revocation сертификата
**Зачем тестируем:** Проверяем, что сервер проверяет статус сертификата (CRL/OCSP) и отклоняет запросы с отозванными сертификатами, даже если токен еще действителен.
